Phase 1 description (concept-first, Markdown):
1. "What is Context Propagation?" — 2-3 sentences. Context propagation is the mechanism that links spans across service boundaries into a single distributed trace. Without it, each service generates an isolated, disconnected trace — you see symptoms but can't follow the request.
2. "How it works" — explain W3C `traceparent` header: the upstream service injects the current trace/span ID into a carrier (HTTP headers, Python dict), the downstream service extracts it and creates a child span. The result: one unified trace across both services.
3. "Why it matters in production" — 1 anecdote: "A checkout request failed after 10s. Without propagated context, auth and payment each showed a normal-looking trace. With propagation, the unified trace revealed auth was waiting 9.8s for payment's connection pool."
4. "The Situation:" — a checkout service calls a payment service. Both are instrumented, but orders are failing and the traces show two disconnected traces instead of one. The context isn't being propagated.
5. "Your Mission:" — Use `inject()` to add the trace context to a carrier dict before calling the downstream service. Use `extract()` in the downstream function to restore the context and create a child span.
6. "Key Concepts:" — traceparent, W3C Trace Context, inject(), extract(), carrier (Python dict simulating HTTP headers).

Validation rules (3):
- `span_exists` — spanName: checkout.process — the upstream span must be created
- `span_exists` — spanName: payment.charge — the downstream span must be created
- `telemetry_flowing` — confirms both spans are captured

Each validation rule must have description, successMessage, errorMessage, hintMessage (starts with "💡 Hint:"), guidedMessage (starts with "📖").

Phase 2 root cause options (4, exactly 1 correct: option b):

const brokenContextRules: RootCauseRule[] = [
  {
    id: 'a',
    label: 'Payment service timing out...',
    specificHint: '💡 Hint: Look at the payment.charge span duration — is it actually slow?',
    evaluate: () => false,
    explainCorrect: () => '',
    explainIncorrect: (data) => {
      const paymentSpan = data.spans.find(s => s.name === 'payment.charge');
      const dur = paymentSpan?.attributes?.['payment.duration_ms'] ?? paymentSpan?.durationMs;
      return `Not quite. The payment.charge span completed in ${dur}ms — not a timeout. The problem is structural: trace.parent_id=${paymentSpan?.attributes?.['trace.parent_id']} shows the payment span is orphaned from the checkout trace.`;
    },
  },
  {
    id: 'b',
    label: 'Checkout service not propagating context...',
    specificHint: '',
    evaluate: (data) => {
      const paymentSpan = data.spans.find(s => s.name === 'payment.charge');
      return paymentSpan?.attributes?.['trace.orphaned'] === 'true';
    },
    explainCorrect: (data) => {
      const paymentSpan = data.spans.find(s => s.name === 'payment.charge');
      return `✓ Correct! The payment.charge span has trace.parent_id=${paymentSpan?.attributes?.['trace.parent_id']} and trace.orphaned=${paymentSpan?.attributes?.['trace.orphaned']} — it was created as a root span instead of a child of checkout.process. Fix: call inject(carrier) before the payment call and extract(carrier) inside charge_payment.`;
    },
    explainIncorrect: () => '',
  },
  {
    id: 'c',
    label: 'Auth service rejecting tokens...',
    specificHint: '💡 Hint: Find the auth span — what is its status?',
    evaluate: () => false,
    explainCorrect: () => '',
    explainIncorrect: (data) => {
      const authSpan = data.spans.find(s => s.name?.includes('auth'));
      return authSpan
        ? `Not quite. There is no auth error in this trace. The checkout.process span shows checkout.step=payment — auth already succeeded.`
        : `Not quite. There is no auth span in this trace at all — auth was not the failing component.`;

  },

    id: 'd',

      const checkoutSpan = data.spans.find(s => s.name === 'checkout.process');
