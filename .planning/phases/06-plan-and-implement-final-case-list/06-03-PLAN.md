```python
from opentelemetry import baggage, context
from opentelemetry.propagate import inject, extract

# Set baggage values — returns a new context (does not mutate current)
ctx = baggage.set_baggage("user.plan", "premium")
ctx = baggage.set_baggage("request.region", "us-east-1", context=ctx)

# Propagate: inject context into carrier
carrier = {}
inject(carrier, context=ctx)

# In downstream: extract context and read baggage
restored_ctx = extract(carrier)
plan = baggage.get_baggage("user.plan", restored_ctx)
```

Phase 1 description (concept-first, Markdown):
1. "What is OpenTelemetry Baggage?" — 2-3 sentences. Baggage is a set of key-value pairs propagated alongside trace context through the entire request chain. Unlike span attributes (which stay in one span), baggage is visible to every downstream service that participates in the same trace.
2. "Baggage vs. Span Attributes" — one clear distinction: span attributes describe what happened in one span; baggage carries information that all services need, like user.plan, feature.flag, or request.region.
3. "Why it matters in production" — anecdote: "A SaaS platform had premium users hitting free-tier rate limits. The rate limiter didn't know the user's plan because the plan wasn't propagated. Baggage solved it in one line."
4. "The Situation:" — an API gateway sets baggage for user.plan and request.region at the entry point, but the downstream rate-limiting service never sees it — premium users are being throttled like free users.
5. "Your Mission:" — Set `user.plan` and `request.region` as baggage in the parent context, propagate via carrier, extract in the downstream function, and annotate the child span with both values.
6. "Key Concepts:" — baggage.set_baggage(), baggage.get_baggage(), propagation (same inject/extract from case 4), carrier.
