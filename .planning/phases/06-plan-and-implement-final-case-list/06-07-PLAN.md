Phase 1 description (concept-first, Markdown):
1. "Congratulations — You've Reached the Capstone" — acknowledge this is the hardest case, requires all previous concepts.
2. "What is a Cascading Failure?" — 2-3 sentences. A cascading failure occurs when one component's degradation causes overload in downstream systems, which in turn causes further failures. Each failure amplifies the next — a small inventory cache miss becomes a checkout outage within seconds.
3. "The Full Observability Stack" — brief recap: spans (case 1-3) tell you what happened; context propagation (case 4-5) links services; metrics (case 6) show trends; logs (case 7) fill the gaps; sampling (case 8) determines what you see. In a cascade, you need ALL of these.
4. "The Situation:" — an e-commerce checkout system has three services: auth-service, inventory-service, and payment-service. They work fine individually, but under load a cascade is occurring that causes checkout failures. You need to add comprehensive telemetry so the cascade is visible.
5. "Your Mission:" — The boilerplate provides all three service functions. You must:
   - Add a span to each service function
   - Propagate context from checkout → auth → inventory → payment
   - Set user.tier as baggage at the entry point
   - Annotate the inventory span with the cache_hit result
   - Annotate the auth span with connection_pool.wait_ms
6. "Key Concepts:" — All concepts from cases 1-8 apply. The diagnostic attributes you add in Phase 1 are what makes Phase 2 solvable.
