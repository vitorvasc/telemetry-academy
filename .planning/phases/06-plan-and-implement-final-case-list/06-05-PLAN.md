<!-- OTel Logging SDK (opentelemetry-sdk, pure Python, no micropip needed) -->
```python
import logging
from opentelemetry.sdk._logs import LoggerProvider
from opentelemetry.sdk._logs.export import SimpleLogRecordProcessor, ConsoleLogExporter
from opentelemetry._logs import set_logger_provider
from opentelemetry.instrumentation.logging import LoggingInstrumentor

# Setup: bridge Python logging to OTel
logger_provider = LoggerProvider()
logger_provider.add_log_record_processor(SimpleLogRecordProcessor(ConsoleLogExporter()))
set_logger_provider(logger_provider)

# Install the bridge — adds trace_id/span_id to all log records
LoggingInstrumentor().instrument(set_logging_format=True)

# Use stdlib logging normally — trace context auto-injected
logger = logging.getLogger(__name__)
logger.info("Processing order", extra={"order_id": "42", "user_id": "usr_9911"})
# When inside a span, this log record automatically has trace_id and span_id
```

Note: `opentelemetry-instrumentation-logging` is part of `opentelemetry-instrumentation` package (pure Python). If not pre-installed, use `micropip.install('opentelemetry-instrumentation-logging')` at top of setup.py.

Phase 1 description (concept-first, Markdown):
1. "What is Structured Logging?" — 2-3 sentences. Structured logging means emitting logs as machine-readable key-value pairs rather than plain text strings. Instead of `"order processed"`, you emit `{"event": "order_processed", "order_id": "42", "user_id": "usr_9911"}`. This enables searching, filtering, and correlation.
2. "Logs + Traces = Full Correlation" — 2-3 sentences. OTel's LoggingHandler automatically injects `trace_id` and `span_id` into every log record emitted inside a span context. This lets you jump from a log line directly to the trace span that generated it — the crucial link for debugging billing errors or auth failures.
3. "Why it matters in production" — anecdote: "A billing failure affected 3 users. The trace showed status: ok — no error at the span level. But the correlated log showed ERROR: billing.charge failed: insufficient_funds for user_id=usr_4821. Without trace correlation, this log would be invisible to the on-call engineer."
4. "The Situation:" — the billing service logs errors but they're plain text with no trace_id field. On-call engineers can't correlate billing logs to the traces where they originated. Three users reported failed charges but the trace viewer shows no errors.

6. "Key Concepts:" — LoggerProvider, LoggingInstrumentor, extra={} for structured fields, trace_id auto-injection, correlation.
