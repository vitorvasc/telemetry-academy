Phase 1 description (concept-first, Markdown):
1. "What is Structured Logging?" — 2-3 sentences. Structured logging means emitting log records as machine-readable key-value pairs rather than plain text strings. Instead of `"charge failed"`, you emit structured fields: `{"user_id": "usr_4821", "billing.attempt": 3, "result": "insufficient_funds"}`. This enables filtering, correlation, and root cause analysis.
2. "Logs + Traces = Full Correlation" — 2-3 sentences. In production observability, the most powerful insight comes from linking a log line to the span that generated it. When spans carry the same structured fields as your logs (user_id, billing.attempt, result), engineers can pivot from a log alert directly to the trace that shows the full request context.
3. "Why it matters in production" — anecdote: "A billing failure affected 3 users. The trace showed status: ok at the API level. But the billing.charge span had billing.attempt=3 and billing.result=insufficient_funds — the same fields that would appear in a structured log. Without these span attributes, the root cause is invisible."
4. "The Situation:" — the billing service spans exist but contain no structured diagnostic fields. On-call engineers can see a billing.charge span but can't tell which user failed, how many attempts were made, or what the rejection reason was.
5. "Your Mission:" — Add structured diagnostic attributes to the billing.charge span: user_id and billing.attempt. These are the same fields that structured logging would emit — but on the span, where they link directly to the trace context.
6. "Key Concepts:" — span attributes as structured log fields, trace correlation, billing.attempt as a retry counter, user_id for user-level debugging.
