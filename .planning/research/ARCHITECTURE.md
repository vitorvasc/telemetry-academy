### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| **EditorPanel** | Captures user's Python instrumentation code | Monaco Editor (`@monaco-editor/react`) |
| **WorkerClient** | Manages Web Worker lifecycle and `postMessage` RPC | React Context + standard JS `Worker` |
| **Pyodide Worker** | Executes Python code safely without blocking UI | `pyodide.js` in a Dedicated Web Worker |
| **OTel JS Exporter** | Bridges Python OTel SDK data back to JS | Custom Python `SpanExporter` calling `js.postMessage` |
| **TelemetryStore** | In-memory sink for spans/logs generated by code | Zustand store (array of spans/logs) |
| **ValidationEngine** | Checks TelemetryStore against Case requirements | Pure functions triggered on store updates |
| **VisualizerPanel** | Renders Trace Waterfall and Log Tables | React components consuming `TelemetryStore` |

## Recommended Project Structure

```
src/
├── core/                  # Engine & Domain Logic
│   ├── worker/            # Web Worker scripts and Pyodide setup
│   │   ├── worker.ts      # Worker entrypoint (handles messages)
│   │   └── otel_setup.py  # Python string injected to setup Custom Exporters
│   ├── store/             # Zustand stores
│   │   ├── useCase.ts     # Current case state, progress, validation
│   │   └── useTelemetry.ts# Captured spans, logs, and trace correlation
│   └── validation/        # Logic to check user's code/telemetry
├── ui/                    # Presentational Components
│   ├── editor/            # Monaco wrapper
│   ├── visualizers/       # Trace waterfall, Log tables
│   └── cases/             # Instructions and hint panels
└── data/                  # Content definition
    └── cases/             # YAML/TS case definitions (Phase 1 & 2 logic)
```

### Structure Rationale

- **core/worker/:** Isolates all the complex asynchronous WASM/Python interop from the React UI. By keeping the python setup string here, we ensure the UI only deals with clean JSON telemetry.
- **core/store/:** Separates the *state* of the game (which phase we are in) from the *data* of the execution (spans and logs).
- **ui/visualizers/:** Reusable pure components that take an array of telemetry data and render it, completely unaware of how the data was generated or verified.

## Architectural Patterns

### Pattern 1: JS-Bound Custom Exporters (The Bridge)

**What:** Instead of using HTTP/gRPC exporters in the Python OTel SDK, we write a custom `SpanExporter` in Python that serializes the data to JSON and invokes a JavaScript callback bound to the Pyodide global scope.
**When to use:** Whenever running backend SDKs in a browser WASM environment.
**Trade-offs:** Avoids CORS, networking complexity, and backend costs. Requires manually translating Python SDK data structures into JSON that matches the UI's expectations.

**Example:**
```python
# Python side (injected before user code)
import json
import js # Pyodide module to interact with JS global scope
from opentelemetry.sdk.trace.export import SpanExporter, SpanExportResult

class BrowserMessageExporter(SpanExporter):
    def export(self, spans):
        for span in spans:
            # Simplify span to dictionary
            span_data = {
                "name": span.name,
                "context": {"trace_id": span.get_span_context().trace_id, ...},
                "attributes": dict(span.attributes) if span.attributes else {}
            }
            # Send back to Web Worker thread, which forwards to Main Thread
            js.postMessage({"type": "TELEMETRY", "dataType": "span", "payload": span_data})
        return SpanExportResult.SUCCESS
```

### Pattern 2: Worker Pre-warming

**What:** Instantiating the Pyodide Web Worker and downloading the `opentelemetry` pip packages in the background *while* the user reads the case instructions.
**When to use:** Always for Pyodide, as `loadPyodide()` and `micropip.install()` can take 2-5 seconds.
**Trade-offs:** Consumes memory and bandwidth immediately on page load, but drastically reduces the perceived latency when the user clicks "Run".

### Pattern 3: Unidirectional Execution Flow

**What:** The UI never mutates telemetry state directly. Code is sent to the worker -> Worker executes -> Worker emits telemetry -> Store updates -> UI Re-renders.
**When to use:** For all code execution lifecycles.
**Trade-offs:** Makes the UI slightly more complex (everything is async), but guarantees that the visualizers only show exactly what the executed code produced.

## Data Flow

### Code Execution & Telemetry Flow

```
[User Clicks "Run Code"]
    ↓
[Editor Component] → [WorkerClient] (postMessage)
                          ↓
[Pyodide Worker] ← [Worker script handles message]
    ↓ (evaluate python)
[OTel SDK Instrumentation] → [BrowserMessageExporter]
    ↓ (intercept spans)
[Worker postMessage] → [Main Thread onmessage]
    ↓
[TelemetryStore] → (Triggers State Update)
    ↓
[Visualizer Panels Re-render] & [Validation Engine Checks Rules]
```

### Key Data Flows

1. **Telemetry Stream:** The user's python code executes synchronously in the worker. As spans open and close, the `BrowserMessageExporter` immediately streams them over `postMessage` back to the React UI, allowing the UI to populate the waterfall in near real-time (or sequentially post-execution).
2. **State Reset Flow:** Every time "Run" is clicked, the `TelemetryStore` is flushed. The Pyodide worker must also reset the OpenTelemetry global `TracerProvider` to ensure spans from previous runs don't leak into the current run.

## Suggested Build Order

To derisk the most complex components first, the implementation phase structure should follow:

1. **Phase A (Infrastructure):** Pyodide Web Worker integration. Establish the `postMessage` loop, load `micropip`, and verify Python `print()` statements stream back to the UI.
2. **Phase B (Observability Bridge):** Install OpenTelemetry via `micropip`. Implement the `BrowserMessageExporter` in Python and wire it to the `TelemetryStore` in React. Verify hardcoded spans show up in the store.
3. **Phase C (Validation Sink):** Connect the `ValidationEngine` to the `TelemetryStore` so that arriving spans automatically satisfy case criteria and unlock Phase 2.
4. **Phase D (Visualization Polish):** Connect the existing mock trace/log viewers to the live `TelemetryStore`.

## Scaling Considerations

| Scale/Constraint | Architecture Adjustments |
|------------------|--------------------------|
| **Worker Initialization** | Use pre-warming (load worker on component mount, not on "Run" click). |
| **Memory Leaks** | Pyodide keeps global state. We must explicitly `sys.modules.clear()` or reset the OTel SDK providers between runs, otherwise memory will balloon after 10-20 code executions. |
| **Main Thread Blocking** | Ensure the UI components processing incoming spans limit re-renders. If thousands of spans are generated (e.g., an infinite loop bug), debounce the `TelemetryStore` updates. |

### Scaling Priorities

1. **First bottleneck:** Pyodide instantiation blocks UI and feels slow. Fix by using Dedicated Web Workers and pre-warming it on initial app load.
2. **Second bottleneck:** Reusing Python global state across multiple executions leaks memory and corrupts telemetry tests. Fix by fully resetting the OpenTelemetry global objects before executing the next run.

## Anti-Patterns

### Anti-Pattern 1: Backend Code Execution

**What people do:** Send the user's Python code to a Docker container on a backend server, collect telemetry, and send it back to the client.
**Why it's wrong:** Introduces massive security risks (Remote Code Execution), requires complex infrastructure, introduces latency, and costs money to scale.
**Do this instead:** Execute strictly inside the browser using WASM (Pyodide).

### Anti-Pattern 2: Re-creating the Worker Per Run

**What people do:** Call `new Worker()` every time the user clicks "Run" to ensure a clean state.
**Why it's wrong:** Re-downloading and re-initializing Pyodide and pip packages takes 3-5 seconds every time, ruining the UX.
**Do this instead:** Keep a long-lived Worker instance. Reset the Python global scope and OpenTelemetry providers programmatically between runs.

### Anti-Pattern 3: Full OTel Backend in Browser

**What people do:** Try to compile Jaeger or Zipkin into WASM to process traces in the browser.
**Why it's wrong:** Over-engineered and heavy. We only need to visualize ~50 spans per case.
**Do this instead:** Parse the simple JSON structures emitted by the custom exporter directly into a lightweight React waterfall/table component.

## Integration Points

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| **React ↔ Worker** | `Worker.postMessage` | Must serialize data. Cannot pass functions or DOM elements. Use strict TypeScript interfaces for message payloads (`{ type: 'SPAN', payload: Span }`). |
| **Python ↔ JS (Worker)** | Pyodide `js` module | Pyodide auto-converts basic types (dicts to Maps, strings to strings). Explicitly serialize complex Python objects to JSON before crossing the boundary to avoid proxy leaks. |
