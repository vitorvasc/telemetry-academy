# Project Research Summary

**Project:** Telemetry Academy
**Domain:** Gamified OpenTelemetry Education Platform (Browser-Based Execution & Visualization)
**Researched:** 2026-02-26
**Confidence:** HIGH

## Executive Summary

Telemetry Academy is an interactive, browser-based educational platform designed to teach OpenTelemetry through a gamified, two-phase loop (Instrument & Investigate). Instead of relying on expensive, complex, and potentially insecure backend infrastructure for executing user-submitted code, experts build this type of product using a client-side WebAssembly (WASM) architecture. Users write Python instrumentation code directly in the browser, which is executed safely in a sandboxed environment.

The recommended approach centers on using Pyodide (Python in WASM) inside a dedicated Web Worker. This setup intercepts the telemetry generated by the official `opentelemetry-sdk` using a custom JS-bound `SpanExporter` and streams the JSON-serialized spans back to a React 19 main thread. The UI then utilizes tools like Zustand for state management and Visx for rendering realistic, D3-style trace waterfalls and log viewers, giving users immediate visual feedback and validation.

The key risks revolve around the constraints of running Python and OpenTelemetry within a browser's WASM environment. Specifically, OpenTelemetry's default `BatchSpanProcessor` will crash due to WASM threading limitations, and passing raw Python objects to JavaScript causes massive memory leaks. These are mitigated by enforcing the `SimpleSpanProcessor`, strictly serializing all telemetry to JSON before crossing the JS bridge, and fully isolating execution in a Web Worker to prevent infinite loops from freezing the UI.

## Key Findings

### Recommended Stack

The stack prioritizes a lightweight, client-side approach that avoids backend code execution while providing a highly interactive UI.

**Core technologies:**
- **React 19 & TypeScript:** UI Framework & Type System — Mandated foundation for concurrent rendering and managing complex telemetry data shapes.
- **Vite:** Build Tool — Essential for fast HMR and handling Pyodide WASM assets.
- **Pyodide:** Client-Side Python — Runs standard `opentelemetry-sdk` directly in the browser via WASM, intercepting telemetry without a backend.
- **@monaco-editor/react:** Code Editor — Provides a VS Code-like experience with Python syntax highlighting and linting.
- **Zustand & @visx/visx:** State & Visualization — Manages cross-component game state and builds custom, lightweight trace flamegraphs.

### Expected Features

**Must have (table stakes):**
- **Interactive Code Editor** — Core mechanic with syntax highlighting.
- **Instant Validation & Feedback** — Checking OTel API usage and outputs immediately.
- **Progressive Disclosure & Persistence** — Case unlocking mechanism and `localStorage` saves.

**Should have (competitive):**
- **Two-Phase Loop** — Teaches both generating (Instrument) and consuming (Investigate) telemetry.
- **Client-Side WASM Execution** — Zero backend infrastructure cost.
- **Realistic Observability UI & Signal Correlation** — Custom UI mimicking real tools, mapping logs to traces via `trace_id`.

**Defer (v2+):**
- **Multi-Language Support** — Sticking to Python for MVP to reduce execution complexity.
- **User Accounts & Case Authoring Tools** — Not essential for validating the core learning loop.

### Architecture Approach

The architecture enforces a unidirectional data flow to isolate complex WASM interop from the React UI.

**Major components:**
1. **Pyodide Web Worker** — Sandboxed execution environment that runs user Python code and the OTel SDK without blocking the UI.
2. **OTel JS Exporter (The Bridge)** — A custom Python `SpanExporter` that serializes span data to JSON and posts it to the JavaScript main thread.
3. **TelemetryStore (Zustand)** — In-memory sink that stores the captured spans and logs, triggering UI re-renders and validation checks.
4. **ValidationEngine & VisualizerPanels** — Pure functions and components that consume the JSON telemetry to verify case requirements and render trace waterfalls.

### Critical Pitfalls

1. **OTel Background Threading in WASM** — Avoid crashes by using `SimpleSpanProcessor` instead of `BatchSpanProcessor`, as Python's `threading` is restricted in WASM.
2. **PyProxy Memory Leaks** — Avoid out-of-memory browser crashes by serializing all telemetry data to JSON in Python *before* posting it to the UI via `postMessage`.
3. **Unkillable Web Workers** — Avoid UI freezes from infinite loops by running Pyodide strictly in a dedicated Web Worker and implementing termination timeouts.
4. **Validation Logic Rigidity** — Avoid frustrating false negatives by validating the *JSON telemetry outputs* rather than parsing the user's Python source code (AST/regex).

## Implications for Roadmap

Based on research, suggested phase structure:

### Phase 1: Execution Sandbox & Infrastructure
**Rationale:** Derisks the most technically complex requirement first: running Python in the browser without UI blocking.
**Delivers:** Pyodide Web Worker integration, Monaco Editor setup, and basic Python execution.
**Addresses:** Interactive Code Editor, Client-Side WASM Execution.
**Avoids:** Unkillable Web Workers (implements dedicated workers and timeouts).

### Phase 2: Observability Bridge & Telemetry State
**Rationale:** Connects the isolated Python execution to the React UI state, enabling telemetry capture.
**Delivers:** Custom Python JS-Exporter, OpenTelemetry SDK installation via `micropip`, and Zustand TelemetryStore.
**Uses:** Pyodide, Zustand.
**Implements:** OTel JS Exporter, TelemetryStore.
**Avoids:** OTel Background Threading (forces `SimpleSpanProcessor`) and PyProxy Memory Leaks (JSON serialization).

### Phase 3: Validation Engine & Core Loop
**Rationale:** Transforms the sandbox into a game by adding rules and feedback based on the telemetry outputs.
**Delivers:** Logic to check store updates against case requirements, unlocking subsequent phases.
**Addresses:** Instant Validation & Feedback, Progressive Disclosure.
**Avoids:** Validation Logic Rigidity (validates JSON spans, not AST).

### Phase 4: Realistic Visualization UI
**Rationale:** Completes the "Investigate" half of the learning loop once real telemetry is successfully captured.
**Delivers:** Visx Trace Waterfall, Log Viewer, and Signal Correlation UI.
**Uses:** @visx/visx, Recharts.
**Addresses:** Realistic Observability UI, Explainable Root Cause Engine.

### Phase 5: Content, Persistence & Polish
**Rationale:** Wraps the core engine in a complete product experience for MVP launch.
**Delivers:** `localStorage` state saving, scenario narratives, hint system, and multiple rookie cases.
**Addresses:** Persistence (Save State), Scenario Narratives, Hint System.

### Phase Ordering Rationale

- **Dependencies:** Web Worker execution must be solved before OTel data can be intercepted. Telemetry data must be successfully intercepted and stored before the validation engine or visualization UI can be built.
- **Risk Mitigation:** Tackling Pyodide and Web Workers in Phase 1 ensures the zero-infrastructure client-side constraint is actually viable before building the rest of the application.

### Research Flags

Phases likely needing deeper research during planning:
- **Phase 4:** Custom Visx trace waterfall rendering can be complex; precise coordinate mapping and hierarchy scaling might require additional prototyping.

Phases with standard patterns (skip research-phase):
- **Phase 1 & 2:** Pyodide Web Worker bridging and custom OTel Exporters are well-documented patterns with clear technical paths.
- **Phase 3 & 5:** Zustand state management and `localStorage` persistence are standard React patterns.

### Gaps to Address

- **Pyodide Initial Load Time:** Fetching the Pyodide runtime and standard library can be slow on poor networks. Needs a pre-fetching strategy during the initial app load or case introduction screens.
- **Complex Attribute Serialization:** Need to ensure custom Python objects or complex dictionaries passed as span attributes are cleanly serialized to JSON and don't break the JS bridge.
